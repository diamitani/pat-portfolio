Overview

ROSTR is Patrick Diamitani's open-source framework for building AI agents and automations that are deep, correct, and durable instead of one-off and forgettable. It unifies four layers: PAL (prompt compilation — setting technical depth and audience framing), NPAO (Navigate / Prioritize / Allocate / Orchestrate — a work-classification spine for any project), RAG/DAL (retrieval + data-access-layer provenance honesty — never presenting an inference as a verified fact), and ContextEngine (a zero-infrastructure, flat-file persistent memory layer so AI sessions stop starting cold). Every skill and agent in this portfolio is built on it. Published as whitepapers and applied in production across a 50+ person GTM org at a global HR-tech (EOR) platform.

The Problem

AI work doesn't compound. Every prompt is written from scratch, every session starts with no memory of the last, quality depends entirely on who's driving, and it's dangerously easy to let a model's confident guess pass as fact. Without a shared methodology, a growing library of skills becomes a pile of inconsistent one-offs that only their author understands — and organizational learning evaporates between conversations.

How It Works

  1. PAL — depth compilation. Before building, PAL sets the dials for the task: how technical the output should be, who the audience is, and how much to translate. It's why a compliance engine and a marketing email built on the same framework each land at the right altitude.
  2. NPAO — work classification. Any project is decomposed into Navigate (research/feasibility), Prioritize (what matters most), Allocate (who/what does what), and Orchestrate (how it runs end to end) — the repeatable spine that becomes a project's "how it works" narrative.
  3. RAG/DAL — provenance honesty. A three-tier credibility rule separates evidenced fact (primary artifacts) from recollection from inference, so nothing gets dressed up as more certain than it is. RAG handles world knowledge; DAL governs how data is accessed and cited.
  4. ContextEngine — the state layer. A flat-file (.context-engine/) persistent memory that survives across sessions: every session compresses into a structured JSON record (what worked, what failed specifically, decisions, open questions), appends to a queryable index, and surfaces the most actionable context at the start of the next session — with zero database or infrastructure.
  5. ROSTR Hub — the assembly. The four layers compose into one operating model: PAL defines what agents are, NPAO determines what they do next, RAG/DAL gives them honest access to knowledge, and ContextEngine gives them memory — so a team of agents behaves like a team, not a set of amnesiac strangers.

Setup Process

  1. Read the ROSTR whitepapers (PAL, NPAO, RAG/DAL, ContextEngine, combined) — the framework is doc-first by design.
  2. Adopt PAL + NPAO as the required shape for any new skill/agent build (this portfolio's skills all follow it).
  3. Drop context-engine.skill into your Claude Code skills directory (or reference it from CLAUDE.md) — it works on first use, no config.
  4. Use the five ContextEngine modes: CACHE ("save this session"), RETRIEVE ("context flash"), REPORT, QUERY, SCHEDULE.
  5. For a multi-agent Hub, wire ContextEngine as the State Layer (auto-cache on session end, auto-retrieve on start); migrate to the Postgres adapter only at enterprise scale.

What You Need

  • Claude Code / Claude access
  • A repo to host .context-engine/ (flat files, ~1MB/year at 10 sessions/week)
  • A willingness to make the methodology the standard, not an option
  • Time-to-adopt: minutes for ContextEngine; ongoing for full-framework adoption

Results

  • Published as an open framework (whitepapers + context-engine.skill), applied to every agent manifest in this portfolio.
  • Turns ephemeral AI sessions into durable, searchable organizational memory — eliminating the "cold start tax" (5–20 min of re-explaining context per session).
  • Provenance honesty baked in: inferences are never presented as evidenced facts, which is what keeps a portfolio's claims defensible under scrutiny.
  • One methodology means any single build in the library reveals the standard behind all of them — the "read one, understand the system" property.

Challenges & Fixes

  • Sessions starting cold, re-discovering the same failures → ContextEngine's append-only session index with specific what_failed entries ("HTTP 401 on token refresh," not "auth issue").
  • Vector-DB overkill for a small iterative team → flat-file memory (no embeddings, no infra) that handles years of sessions before any scaling concern.
  • Models overstating certainty → RAG/DAL three-tier provenance rule enforced at the framework level.
  • Inconsistent, author-dependent builds → PAL/NPAO as a shared, published spec every new skill inherits.

Framework/architecture thinking · provenance discipline in AI systems · zero-infrastructure engineering pragmatism · building for compounding, not one-offs.
