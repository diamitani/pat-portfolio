The Challenge

Chats ended. Leads evaporated.

A website chat window is where buying intent is hottest and attention is shortest. Atlas's site needed a front-line agent that could greet visitors, answer questions about pricing, country coverage, payroll, and compliance — and, critically, not let a qualified buyer evaporate when the chat window closed.

The harder problem was what happens after the conversation. A transcript sitting in a shared inbox is dead data: nobody re-reads closed chats, nobody manually creates contacts from them, and nobody can say whether the chatbot program is generating revenue or just answering questions. Without a qualification-and-capture layer, even a great conversation was a lead leak.

The approach: every closed conversation is a lead-scoring event

Rather than qualifying in real time — intrusive and error-prone mid-chat — the pipeline waits until a conversation ends, then has an AI read the entire transcript with full context and make one considered judgment: is this person a Marketing Qualified Lead?

Attribution followed the same conservative standard as the Inbound Automation study: a deal counts only if the contact originated through the chatbot and the deal was created after their first chat. A $250K deal the chatbot touched — but which arrived through the inbound form first — stays credited to the other channel. No double counting between the two programs.

What Was Built

A 14-node close-conversation pipeline.

Maia handles the conversation; the pipeline makes it count — conversation → transcript → LLM judgment → CRM action → revenue attribution.

01 · Trigger

Close-event webhook

A webhook fires whenever a chatbot conversation is closed in HubSpot Inbox.

02 · Retrieval

Conversations API traversal

HTTP nodes walk the HubSpot Conversations API — thread → messages → contact — and a code node normalizes it all into a clean, timestamped transcript.

03 · Judgment

Structured AI verdict

An Azure OpenAI agent with a structured output parser reads the full conversation and returns strict JSON: MQL / Non-MQL, a confidence score, and the target lifecycle stage.

04 · CRM Action

Contact enriched, not summarized

HubSpot nodes write the verdict, archive the full transcript to the contact record, and set the lifecycle stage — sales sees the whole story, not a summary of a summary.

05 · Segmentation

Lists & dashboards

HubSpot lists (Chatbot Contacts / MQLs / Deals) plus a native workflow segment everything for reporting — instrumented from day one.

06 · Proof

Measurement pipeline

Same execution-level methodology as the inbound study: complete 404-run history pulled from n8n, parsed, matched to CRM deals via HubSpot MCP, modeled in a formula-driven workbook.

Systems Thinking

Architected so AI mistakes can't destroy value.

  • Capture beats classification. The $7M buyer's conversation was AI-ruled Non-MQL on July 12 — but the chatbot had already captured the contact on July 8, and sales created the deal on July 14. The system is layered so a conservative classifier can't lose a captured lead: contact, transcript, and human follow-up exist independently of the verdict.
  • Qualify at close, not mid-chat. Scoring the finished transcript gives the AI full context and keeps the visitor experience clean — a deliberate tradeoff against real-time routing speed.
  • Errors as filters. 267 of 404 runs "errored" — overwhelmingly contact-lookup failures on anonymous visitors who never shared an email. That's the pipeline refusing to fabricate CRM records for ghosts; the error column separates real defects (19 worth fixing) from by-design filtering.
  • Cross-program attribution discipline. One deal was touched by both the chatbot and the inbound form automation. One rule — first-originating channel with deal-created-after — assigns it once, and both case studies state the overlap explicitly.
  • Instrument from day one. Segments, dashboards, and per-conversation verdicts were built alongside the bot, so "is this working?" was answerable from the first week.
